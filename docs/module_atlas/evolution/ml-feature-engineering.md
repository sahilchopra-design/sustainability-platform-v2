## 9 · Future Evolution

### 9.1 Evolution A — Run the real pipeline on platform data instead of describing one (analytics ladder: rung 1 → 2)

**What.** §7's flag is complete: the page's footer *claims* KSG-estimator mutual information and SVD-based PCA verbatim, but "MI Score" is a relabelled linear-decay `importance` draw, the 20×20 "correlation matrix" is independent PRNG cells with no symmetry or positive-semi-definiteness, the scree plot is a fixed exponential (`15·e^{−0.08i}`) with hand-typed "85% at 50 PCs" prose, the `topN` slider is cosmetic, and `DQ_IMPACT`'s RMSE/R² numbers are typed, not measured. Only 20 of the claimed 948 features exist as names. Evolution A makes the claims true at honest scale: the platform genuinely holds company-level ESG score panels (`GLOBAL_COMPANY_MASTER`, sector benchmarks, emission intensities), and sklearn is already in the environment per the platform's rung-4 tooling convention — so compute *actual* PCA eigendecomposition, *actual* MI estimates and a *valid* correlation matrix over the real feature panel, with velocity/acceleration features derived from whatever score history exists (honest about panel depth: if only 2–3 periods exist, say so and compute what's computable).

**How.** (1) A backend job (`POST /ml-features/compute`) assembling the feature matrix from platform tables, running standardised PCA and MI-vs-target (target = a declared platform outcome variable, e.g. ESG-score change), persisting results with data vintage. (2) The catalog reflects the real assembled features with measured completeness per column — data-quality impact becomes a measured ablation (retrain with DQ-tier subsets), replacing the typed `DQ_IMPACT` table. (3) The footer's methodology text becomes true or is corrected. (4) Matrix validity checks (symmetry, PSD) as unit tests — the current defect class made impossible.

**Prerequisites.** A declared target variable and feature-panel assembly query; the seeded catalog deleted; scope honesty about 948-vs-actual feature count. **Acceptance:** the scree plot changes when the input panel changes; the correlation matrix is symmetric PSD; MI rankings recompute against the declared target; every headline claim in the UI is measured.

### 9.2 Evolution B — Feature-store explainer for model builders (LLM tier 2)

**What.** Once the pipeline is real, its consumers (the sibling `ml-risk-scorer` and any platform model work) need a feature-store copilot: "which features carry the most information about ESG-score deterioration, and how correlated are the top 10?", "what does dropping DQ4/DQ5 data do to model error?" (the measured ablation answers this), "why did PC1's loading shift after the last data refresh?" — questions over computed artifacts with vintage comparisons.

**How.** Tier 2 over the compute-results routes: every statistic quoted carries its computation date and input-panel vintage; loading/importance explanations reference the actual eigenvector or MI values from stored results; drift observations ("PC1 rotated toward emissions intensity") compare stored runs mechanically. The copilot's hard rule mirrors this module's cautionary history: no statistical claim without a stored computation behind it — the page once asserted "no degradation, 85% variance" as static prose, and the copilot must be architecturally incapable of that failure (the no-fabrication validator checks every number against the results store). Methodology questions (why KSG for MI, why standardise before PCA) ground in the corrected documentation.

**Prerequisites (hard).** Evolution A's computed results store; Phase 2 tooling. **Acceptance:** every figure resolves to a stored run with vintage; cross-run comparisons cite both run ids; methodology answers match the implemented (not aspirational) pipeline.
